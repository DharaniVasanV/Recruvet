# Fake Job Post System - Feature Testing Guide

This guide explains how to verify each major feature after local setup or Render deployment.

## 1. Pre-Test Checklist

Before testing, confirm these are ready.

- Java 17 is installed.
- Maven is installed.
- PostgreSQL database is reachable.
- Required environment variables are configured.
- Render deployment has completed successfully, or local app is running.
- Google OAuth redirect URLs are configured.
- Tesseract is installed locally, or installed through Docker on Render.
- `models/fake_job_model.pkl` and `models/tfidf_vectorizer.pkl` are present if ML model scoring should use the trained model.

Run this command before manual testing:

```powershell
mvn test
```

Expected result:

```text
BUILD SUCCESS
```

## 2. Database Verification

After the app starts once, Hibernate should create or update the required tables.

Run this SQL in pgAdmin or Render PostgreSQL:

```sql
SELECT table_name
FROM information_schema.tables
WHERE table_schema = 'public'
ORDER BY table_name;
```

Expected important tables:

```text
users
predictions
institution
batch_job
outcome_report
company_reputation_aggregate
```

Insert at least one institution for TPO signup:

```sql
INSERT INTO institution (name, verified_email_domain, subscription_tier)
VALUES ('Sri Eshwar College of Engineering', 'sece.ac.in', 'FREE');
```

Verify:

```sql
SELECT * FROM institution;
```

## 3. Authentication Testing

### Normal Signup

Steps:

1. Open `/signup`.
2. Enter a normal email and password.
3. Choose `Student / Individual`.
4. Submit.

Expected result:

The account is created and the user is redirected to `/dashboard`.

Database check:

```sql
SELECT email, role FROM users WHERE email = 'your-email@example.com';
```

Expected role:

```text
ROLE_USER
```

### Google Login

Steps:

1. Open `/login`.
2. Click `Sign in with Google`.
3. Select Google account.

Expected result:

User is redirected to `/dashboard`.

If it fails, verify Google Cloud Console:

```text
Authorized JavaScript Origin:
http://localhost:8081
https://your-render-app.onrender.com
```

```text
Authorized Redirect URI:
http://localhost:8081/login/oauth2/code/google
https://your-render-app.onrender.com/login/oauth2/code/google
```

## 4. Main Job Analysis Testing

### Text Input Analysis

Steps:

1. Login.
2. Go to dashboard.
3. Paste a job description.
4. Click `Analyze Job Post`.

Expected result:

The result page should show:

- Total risk score.
- ML model score.
- Review reputation score.
- AI verification score if Groq is configured.
- Company verification.
- Extracted evidence trail.
- Analyzed job description preview.

Database check:

```sql
SELECT company_name, score, ml_score, review_risk_score, groq_risk_score
FROM predictions
ORDER BY id DESC
LIMIT 5;
```

### Image OCR Analysis

Steps:

1. Login.
2. Upload an offer letter image.
3. Click `Analyze Job Post`.

Expected result:

The result page should show extracted text, company details, contact emails, and risk score.

If OCR fails:

- Local Windows: check `APP_TESSERACT_COMMAND` points to `tesseract.exe`.
- Render: check `APP_TESSERACT_COMMAND=tesseract`.

### PDF Analysis

Steps:

1. Login.
2. Upload a PDF offer letter.
3. Click `Analyze Job Post`.

Expected result:

The app extracts PDF text and shows risk analysis. For document-like inputs, the forensics section should appear.

## 5. ML Model Testing

The ML score is generated by:

```text
scripts/predict_score.py
models/fake_job_model.pkl
models/tfidf_vectorizer.pkl
```

Manual script test:

```powershell
"urgent job whatsapp pay registration fee" | python scripts/predict_score.py
```

Expected result:

A numeric score such as:

```text
0.734211
```

If the script fails, Java falls back to heuristic scoring in `MlInferenceService`.

## 6. Entity Extraction Testing

Use a job post containing:

- Company name.
- Email.
- Website.
- Contact person.
- Phone number.

Expected result:

The result page should show extracted organizations, contact persons, emails, and company verification links.

Services involved:

- `EntityExtractionService`
- `GeminiEntityCleanupService`
- `CompanyLookupService`

## 7. Review Reputation Testing

This feature uses Serper search, scraping, and Gemini analysis.

Required keys:

```text
SERPER_API_KEY
GEMINI_API_KEY
```

Steps:

1. Analyze a post with a known company name.
2. Open browser console.
3. Check review evidence logs.

Expected result:

The result page should show:

- Review reputation score.
- Positive and negative review signal counts.
- Review evidence sources.
- Evidence trail entries.

Console may show:

```text
Review Evidence Types
Reddit: Scraped
Indeed: Fallback
Glassdoor: Scraped
```

## 8. Groq AI Verification Testing

Required key:

```text
GROQ_API_KEY
```

Steps:

1. Analyze a suspicious offer letter.
2. Check result page.

Expected result:

The `AI Deep Verification Report` should appear with:

- AI risk score.
- Summary.
- Red flags.

## 9. Evidence Trail Testing

Steps:

1. Run any analysis.
2. Open the `Evidence Trail` section.

Expected categories:

```text
ML
REVIEW
GROQ_AI
FORENSICS
DOMAIN_AGE
EMPLOYEE_MISMATCH
```

Not all categories appear for every input. They appear only when that signal exists.

## 10. Outcome Report Testing

Steps:

1. Analyze a job.
2. Scroll to `Did you find out what happened with this?`
3. Submit outcome as `Confirmed scam`, `Confirmed legitimate`, or `Unsure`.

Expected result:

The report is saved as pending.

Database check:

```sql
SELECT company_name, confirmed_status, moderation_status
FROM outcome_report
ORDER BY submitted_at DESC;
```

Expected moderation status:

```text
PENDING
```

## 11. Admin Moderation Testing

First make a user admin:

```sql
UPDATE users
SET role = 'ROLE_ADMIN'
WHERE email = 'your-email@example.com';
```

Steps:

1. Login as admin.
2. Open `admin/outcome-reports`.
3. Approve or reject pending reports.

Expected result:

Approved reports update community reputation aggregates.

Database check:

```sql
SELECT * FROM company_reputation_aggregate;
```

## 12. Community Warning Testing

Requirement:

At least 3 approved `CONFIRMED_SCAM` reports from distinct users for the same normalized company.

After that, future analyses for that company should show:

```text
Community Confirmed Outcome Warning
```

This warning is displayed separately and is not silently added into the weighted risk score.

## 14. TPO CSV Batch Testing

CSV format:

```csv
company name,posting text or URL
Example Tech,"Urgent hiring. Pay registration fee. Contact on WhatsApp."
Safe Company,"Software engineer role with normal interview process."
```

Steps:

1. Login as TPO.
2. Upload CSV from `/tpo/dashboard`.
3. Wait while batch rows process asynchronously.

Expected result:

- Batch job appears in recent batch jobs.
- Processed row count increases.
- Predictions appear under institution postings.
- High-risk rows show warning badges.
- Repeat offenders are highlighted if present in aggregate database.

## 15. WhatsApp Testing

Required keys:

```text
WHATSAPP_ACCESS_TOKEN
WHATSAPP_PHONE_NUMBER_ID
WHATSAPP_VERIFY_TOKEN
APP_PUBLIC_BASE_URL
```

Meta webhook URL:

```text
https://your-render-app.onrender.com/webhook/whatsapp
```

Steps:

1. Configure webhook in Meta Developer Dashboard.
2. Use the same verify token as `WHATSAPP_VERIFY_TOKEN`.
3. Send a text job post to the test WhatsApp number.
4. Send an image offer letter to the test WhatsApp number.

Expected result:

The app replies with:

- Low/Medium/High risk verdict.
- Top flags.
- Public result link.

Development note:

Meta test numbers work only for manually verified recipient phone numbers until production approval.

## 16. Render Deployment Testing

Environment values on Render:

```text
DATABASE_URL
GOOGLE_CLIENT_ID
GOOGLE_CLIENT_SECRET
APP_PYTHON_COMMAND=python3
APP_TESSERACT_COMMAND=tesseract
APP_UPLOAD_DIR=/tmp/uploads
GEMINI_API_KEY
SERPER_API_KEY
GROQ_API_KEY
APP_PUBLIC_BASE_URL=https://your-render-app.onrender.com
```

Render expected behavior:

- App starts without database URL parse error.
- Server binds to Render `PORT`.
- Login page loads with CSS.
- Google login works after redirect URL update.
- Text analysis works.
- PDF/image analysis works if Tesseract is installed in Docker image.

## 17. Known Optional Features

These can stay disabled during basic testing:

- WhatsApp intake.
- TPO weekly email digest.
- Whois domain age check.

Disable weekly digest:

```text
TPO_DIGEST_ENABLED=false
```
